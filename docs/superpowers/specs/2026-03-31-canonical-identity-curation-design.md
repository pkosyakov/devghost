# Design: Canonical Identity Curation (Slice 5A)

## Summary

Make canonical contributor identity the only layer the customer reviews and edits. Remove manual raw developer dedup from the analysis setup path. Keep contributor selection as a lightweight scope control. Preserve billing safety.

This is the target-state identity solution, not a temporary bridge.

## Target Flow

```text
Create order (DRAFT)
  -> auto-extract contributors (-> DEVELOPERS_LOADED)
  -> contributor selection UI (include/exclude, NO merge/dedup)
  -> estimated credits update per scope
  -> Analyze (-> PROCESSING)
  -> results (COMPLETED)
  -> People for identity review
```

Removed from default path:
- dedup groups UI
- merge/unmerge in analysis setup
- save mapping as a mandatory step

Retained:
- contributor extraction (machine-generated)
- include/exclude selection
- billing from scoped contributors

## Status Flow

```text
DRAFT -> DEVELOPERS_LOADED -> PROCESSING -> COMPLETED
```

`READY_FOR_ANALYSIS` is no longer a required gate. The analyze route already accepts orders not in PROCESSING status. If removing `READY_FOR_ANALYSIS` causes test/code breakage, it can be kept temporarily but must not be a blocking step requiring manual dedup.

## Billing Preflight

Billing is derived from analysis scope, not from a manual dedup workflow.

Analysis scope consists of:
- selected repositories
- selected period (ALL_TIME, DATE_RANGE, LAST_N_COMMITS)
- selected contributors (include/exclude)
- cache hits

Flow:
1. `selectedDevelopers` is machine-generated by the extraction endpoint (already happens today)
2. User toggles include/exclude on contributor selector
3. `excludedDevelopers` is persisted
4. Analyze route sums `commitCount` from non-excluded `selectedDevelopers`
5. Subtracts cache hits
6. Reserves credits conservatively
7. Launches analysis
8. Unused credits released after completion

No change to the actual billing math. The change is that billing no longer requires a manual dedup/mapping step as a prerequisite.

## Compatibility Artifacts

### selectedDevelopers

Remains as **machine-generated contributor candidates**. Generated by the extraction endpoint automatically. Contains identity signals: email, name, login, avatarUrl, commitCount, repositories.

The user does not author or edit this data. They only include/exclude contributors.

### developerMapping

**Empty by default** in the new flow. No trivial mapping generated unless a specific legacy path is confirmed to break without it.

Rationale: `extractAliasesFromOrder()` in `contributor-identity.ts` already works from `selectedDevelopers` alone when `developerMapping` is empty or absent. The mapping was only needed for merged groups from manual dedup, which no longer happens in the default path.

If a legacy path (e.g., worker metric grouping) is confirmed to require non-empty mapping, generate a trivial 1:1 mapping as a machine-generated compatibility artifact, not as a domain concept.

## Canonical Identity Projection

### Source

Machine-generated extracted contributor candidates from `selectedDevelopers`.

NOT from `CommitAnalysis` only. `CommitAnalysis` contains only `authorEmail` and `authorName`, which would lose GitHub login, avatar, and provider signals needed for quality identity resolution.

### Projection Logic

For each candidate in `selectedDevelopers`:
1. Create/update `ContributorAlias` with available signals (email, login, username, avatarUrl)
2. Auto-resolve using deterministic matching only:
   - **Exact providerId match** (if extractor is enriched with providerId — not available today)
   - **Exact email match** -> AUTO_MERGED
   - **Same GitHub login** as deterministic provider-level match -> AUTO_MERGED
3. No providerId match and no email/login match -> UNRESOLVED -> identity queue
4. **No fuzzy matching** (no Levenshtein, no domain similarity, no initial+lastname)

### Identity Signal Availability

Current extraction payload provides:
- `email` (always present)
- `name` (always present)
- `login` (present when GitHub author is resolved)
- `avatarUrl` (present when GitHub author is resolved)
- `commitCount`, `repositories` (always present)

NOT currently available:
- `providerId` (GitHub numeric user ID) — must be preserved if added to extractor

Current minimum deterministic signals: **email + login**.

### What Changes in `projectContributorsFromOrder()`

- Remove dependency on `developerMapping` (merged_from groups)
- Read `selectedDevelopers` directly as flat alias seed
- Each developer -> one alias, no merge topology
- Same auto-resolution logic (providerId match, email match, create Contributor)
- `CommitAnalysis` used as evidence layer (lastSeenAt, commit scope), not as primary identity source

## UI Changes

### Order Detail — DEVELOPERS_LOADED State

Replace the current dedup/merge screen with a clean contributor selector:

- Flat list of contributors (name, email, avatar, commitCount, repos)
- Include/exclude checkboxes (all included by default)
- Estimated credits display (updates dynamically on toggle)
- "Start Analysis" CTA
- Search/filter contributors

NOT shown:
- Dedup groups
- Merge suggestions
- Confidence badges
- Save mapping button

### Order Detail — COMPLETED State

- Analysis results as today
- If unresolved aliases exist: banner "N contributors need identity review" linking to People
- Clear handoff to People for identity trust work

### Legacy Dedup UI

Removed from the default path. If retained for internal/admin use, accessible via query param (`?advanced=true`) or similar mechanism. Not part of the standard onboarding journey.

| Analyze route | `api/orders/[id]/analyze/route.ts` | Accept from DEVELOPERS_LOADED, persist excludedDevelopers inline, auto-generate mapping if needed |

## Hard Rules

- Do not weaken credit estimation or reservation correctness
- Do not auto-merge fuzzy matches into canonical contributors
- Do not remove manual merge/unmerge capabilities (they live on People/canonical surfaces)
- Do not keep raw developer dedup as the default onboarding path
- Do not force customers to pay for all discovered contributors if they selected only a subset
- Do not make new identity surfaces depend on order-local JSON as the business contract
- Do not degrade provider/login-based identity quality

## Acceptance Criteria

- [ ] New customer can run analysis without mandatory raw developer dedup step
- [ ] Billing reserved safely before analysis for selected repo + period + contributor scope
- [ ] User can include/exclude contributors before analysis and see price impact
- [ ] Canonical contributor projection no longer depends on customer-authored manual dedup
- [ ] Unresolved/suggested identity review available through People / identity queue
- [ ] Manual merge/unmerge still works on canonical surfaces
- [ ] Typecheck passes
- [ ] Relevant tests pass

## Risks and Follow-ups

- **developerMapping empty**: Need to verify all worker/metric paths handle empty mapping gracefully. If something breaks, generate trivial 1:1 mapping as compatibility shim.
- **providerId enrichment**: Current extractor does not include GitHub numeric user ID. Adding it would improve auto-resolution quality. Can be done in this slice or as immediate follow-up.
- **READY_FOR_ANALYSIS status**: May have downstream consumers (tests, admin views). Removal should be verified or deferred to cleanup.
- **Legacy dedup access**: Decision on whether to keep behind advanced flag or remove entirely.
- **Cross-order cache**: Current cache logic may reference selectedDevelopers. Need to verify it works with the new flow.
