## 1. Постановка проблемы

Текущий пайплайн FD v2 для крупных коммитов (50+ файлов) систематически переоценивает трудозатраты в 2-10 раз. Валидация на 6 ground truth коммитах показала MAPE 80.4%, цель — менее 50%.

**Анализ корневых причин** (валидация на claude-sonnet-4, 6 GT-коммитов):

| Проблема | Доказательства | Влияние |
|----------|---------------|---------|
| Branch B (покластерная LLM-оценка) переоценивает в 3-10 раз | Каждый кластер оценивается изолированно, теряя контекст целого коммита | Сумма 15 кластеров = 128-457ч при GT 15-60ч |
| Branch A (полный дифф в одном вызове) не помещается | Даже отфильтрованные диффы — 130K-1M токенов | Автоматически откатывается на Branch B, реально не используется |
| heuristic_total добавляется поверх LLM-оценки | combine_estimates() прибавляет часы тривиальных файлов после LLM-оценки | Раздувает 16dc74e с 8.5ч (идеальный holistic) до 30.2ч |
| Holistic-оценка неожиданно точна | 2/5 в GT-диапазоне, 5/5 в пределах 2x — при минимальных метаданных | Хороший сигнал тонет под сломанными компонентами |

**Данные валидации (v2, Branch B + Holistic + combine):**

| Коммит | Branch B | Holistic | Итого (v2) | GT-диапазон | Точность holistic |
|--------|---------|----------|-----------|-------------|-------------------|
| 1d02576 dialer (272 файла) | 456.7ч | 85.0ч | 88.7ч | 40-60ч | 1.4-2.1x |
| 16dc74e vitest (1036 файлов) | 43.5ч | 8.5ч | 30.2ч | 8-16ч | **в диапазоне** |
| 9c2a0ed visitors (159 файлов) | 231.2ч | 45.2ч | 53.6ч | 25-40ч | 1.1-1.8x |
| 18156d0 temporal (123 файла) | 156.4ч | 28.5ч | 49.2ч | 20-35ч | **в диапазоне** |
| 7d4a37e chat (105 файлов) | 128.2ч | 28.5ч | 34.8ч | 15-25ч | 1.1-1.9x |

**Ключевой вывод**: Holistic-оценщик — видя только метаданные, без кода — оказался самым точным компонентом. Академическая литература (COCOMO II, COSMIC, JIT-SDP) подтверждает: чтение кода не требуется для оценки трудозатрат. Сигнал — в структуре, а не в содержании.

## 2. Предлагаемый подход: один холистический вызов с богатыми метаданными

### 2.1 Основная идея

Заменить двухветочную архитектуру (Branch A + Branch B + holistic + combine) **одним LLM-вызовом**, который получает **богатый структурированный блок метаданных** и выдаёт одну оценку. Код (дифф) не передаётся. heuristic_total не добавляется отдельно — LLM видит полную картину и выдаёт одно число.

### 2.2 Почему это работает

1. **Нет переполнения контекста** — блок метаданных ~2-4K токенов независимо от размера коммита
2. **Контекст целого коммита сохранён** — LLM видит всю структуру, а не изолированные куски
3. **Нет двойного счёта** — одна оценка покрывает всё, без суммирования heuristic + LLM
4. **Валидировано COCOMO II** — самая успешная параметрическая модель использует ноль инспекции кода
5. **Эмпирически продемонстрировано** — наш holistic-компонент уже дал 2/5 попаданий в диапазон с минимальными метаданными

### 2.3 Что меняется относительно v2

| Компонент | v2 (текущий) | v3 (предлагаемый) |
|-----------|-------------|-------------------|
| Классификация файлов | SKIP / HEURISTIC / LLM_REQUIRED | Та же классификация, но роль меняется — **подаёт метаданные в промпт** |
| Heuristic_total | Прибавляется к LLM-оценке | **Передаётся как метаданные LLM**, не прибавляется отдельно |
| Branch B (кластерная LLM) | 15 LLM-вызовов на диффах кластеров | **Удаляется** |
| Branch A (полный дифф LLM) | 1 LLM-вызов на отфильтрованном диффе | **Удаляется** |
| Holistic (метаданные LLM) | 1 LLM-вызов, минимальные метаданные | **Усиливается**: 1 вызов, богатый блок метаданных |
| Combine | min/avg ветки + holistic + heuristic | **Не нужен** — одна оценка |
| LLM-вызовов на коммит | 16-31 вызовов | **1 вызов** (+ опциональный judge) |
| Стоимость за коммит | ~$0.30-0.80 (sonnet, 15 кластеров) | **~$0.01-0.03** |

## 3. Богатый блок метаданных — набор фич

На основе приоритизированного набора фич из исследования, привязанного к тому, что наш пайплайн уже вычисляет.

### 3.1 Фичи, которые уже есть (бесплатно)

Вычисляются в `_tag_file()`, `classify_file_regex()`, `adaptive_filter()` и `run_fd_hybrid()`:

**P0 — Базовые структурные:**
- `fc`, `la`, `ld` — файлов изменено, строк добавлено, строк удалено
- `new_file_ratio` — % полностью новых файлов (прокси для COCOMO RUSE multiplier)
- Распределение по расширениям — уже вычисляется в `_build_metadata_prompt()`
- Распределение тегов — `test`, `generated`, `config`, `docs`, `locale`, `test_data` и т.д.
- Поимённая статистика — имя файла, строк +/-, теги (первые 50 файлов)

**P1 — На основе паттернов:**
- Детекция перемещений/переименований — `classify_move_commit()` -> `is_move`, `move_type`, `pairs`
- Массовый рефакторинг — `detect_bulk_refactoring()` -> `is_bulk`, `bulk_ratio`, `pattern_description`
- Сигналы scaffold — `has_scaffold_signal`, `is_bulk_new`, `is_near_total_add`
- Исключение сгенерированных файлов — `classify_file_tier()` SKIP-тир

### 3.2 Фичи для добавления (почти бесплатно)

**P0 — Отсутствуют в текущем пайплайне:**

1. **Энтропия распределения изменений**
   ```
   H = -sum(p_i * log2(p_i))  где p_i = lines_changed_i / total_lines_changed
   ```
   Высокая энтропия = изменения распределены равномерно (систематический рефакторинг). Низкая энтропия = сконцентрированы (целевая фича). Исследования: сильнейший предиктор после churn в мета-анализах JIT-SDP. Стоимость: 5 строк кода.

2. **Эффективный churn** (после исключения сгенерированных файлов)
   Мы считаем `heuristic_total` для отфильтрованных файлов, но не сообщаем `effective_la` / `effective_ld` — строки только в содержательных файлах. Это "реальный" размер, о котором должна думать LLM.

3. **Распределение размеров файлов** — p50, p90, max строк на файл
   Различает "100 файлов по 10 строк" от "5 файлов по 2000 строк". Стоимость: одна сортировка + индекс.

4. **Количество модульных границ**
   Количество уникальных директорий верхнего уровня = архитектурная широта. Стоимость: set comprehension по путям.

### 3.3 Фичи через дешёвый LLM-предвызов (Фаза 2)

5. **Классификация intent** — feature / bugfix / refactor / migration / tooling / churn
   Claude Haiku, 1 вызов, <$0.001. Исследования указывают, что это сильнейший семантический сигнал. Уточнение: точность 100% XGBoost из исследования — для бинарной классификации refactoring vs non-refactoring, не для полной 6-классовой таксономии. Для общей классификации точность будет ниже.

6. **Архитектурный охват** — isolated / cross-module / system-wide
   Можно приблизить по структуре директорий без LLM или дополнить LLM-вызовом.

### 3.4 Отложенные фичи (Фаза 3+)

7. Дельта AST-сложности через tree-sitter
8. Байесовский prior (эвристика в стиле COCOMO) + LLM posterior с обучаемыми весами

## 4. Архитектура промпта

### 4.1 Системный промпт

```
You are an expert software effort estimator. Estimate how many hours a
MID-LEVEL developer (3-5 years experience, NO AI copilot) would need to
implement the described changes.

CALIBRATION (starting heuristics — will be revised after first 10-20 GT commits):
- 1 hour = approximately 50-100 lines of non-trivial logic for a mid-level dev
- Generated files (lock, .d.ts, protobuf, snapshots) = 0 hours
- Renamed/moved files = 0.5h per 50 files of restructuring
- Test code = approximately 50-75% effort of corresponding logic code
- Config files (tsconfig, eslint, docker) = 0.1-0.5h each
- Documentation = 0.3h per 100 lines
- Bulk same-edit refactors (import rename across 200 files) = 2-4h total

ANTI-OVERESTIMATION:
- Large file counts DO NOT mean large effort. Most big commits are
  dominated by generated code, migrations, config, or bulk renames.
- A 500-file commit is often 8-20h, not 100h+
- New files that are boilerplate/scaffold are cheap (0.1h each)
- Only genuinely novel algorithm/business logic code is expensive

RESPONSE FORMAT (JSON):
{"low": N, "mid": N, "high": N, "confidence": "low|medium|high", "reasoning": "..."}

Where:
- low/mid/high = estimated hours range
- confidence = your certainty level
- reasoning = 2-3 sentences: (1) change type classification, (2) effective code size, (3) complexity adjustment
```

### 4.2 Пользовательский промпт — структурированный блок метаданных

```
COMMIT: {message}
LANGUAGE: {language}

CHANGE VOLUME:
- Raw: {fc} files, +{la}/-{ld} lines
- After filtering generated/trivial: {effective_fc} substantive files, +{effective_la}/-{effective_ld}
- Generated/auto (0h): {skip_count} files
- Trivial (config/test/docs): {heuristic_count} files (~{heuristic_total:.1f}h by formula)

FILE TYPE BREAKDOWN:
- Logic: {logic_count} files, +{logic_lines} lines
- Tests: {test_count} files, +{test_lines} lines (test ratio: {test_ratio}%)
- Config/infra: {config_count} files
- Documentation: {docs_count} files
- New files (from scratch): {new_count} ({new_pct}% of total)

DISTRIBUTION:
- Entropy: {entropy:.2f} (max={max_entropy:.2f}; {interpretation})
- Largest file: {max_file} (+{max_lines} lines)
- File sizes: p50={p50} lines, p90={p90} lines
- Modules touched: {module_count} ({module_list})

PATTERN FLAGS:
{flags_or_none}

STRUCTURE (substantive files grouped by directory):
{cluster_summary}

Estimate total hours for implementing this entire commit.
```

### 4.3 Ключевые проектные решения

**heuristic_total ПОКАЗЫВАЕТСЯ как метаданные, НЕ прибавляется отдельно.**

Промпт показывает: `Trivial (config/test/docs): 370 files (~21.7h by formula)`. LLM сама решает, каков реальный вес. Для vitest-миграции (370 переименований jest->vitest конфигов) LLM должна распознать тривиальность и снизить до ~2ч. Для реальной фичи с 30 файлами тестов она может оценить 15ч. У LLM есть семантический контекст для суждения — у слепой формулы его нет.

Это напрямую исправляет главный убийцу точности в v2: коммит 16dc74e, где holistic идеально попал в 8.5ч, но был раздут до 30.2ч добавлением 21.7ч heuristic.

**Структура кластеров — только метаданные, не покластерная оценка.**

Мы по-прежнему запускаем `build_clusters()` для организации файлов в логические группы. Но вместо отправки диффа каждого кластера в LLM мы показываем имена кластеров, количество файлов и строк как метаданные. Это даёт архитектурный контекст: "3 кластера в `api/services/`, 2 в `web/components/`, 1 в `migrations/`" — достаточно для понимания формы работы без чтения кода.

**3-шаговый constrained CoT через поле `reasoning`.**

Исследования показывают, что ограниченный CoT (принудительные аналитические шаги) превосходит и открытый CoT, и прямой ответ для структурированных задач оценки. Три шага: (1) классифицировать тип изменения, (2) оценить эффективный объём кода, (3) скорректировать на сложность/новизну.

**Диапазоны (`low`, `mid`, `high`), а не точечные оценки.**

Исследования: запрос диапазонов даёт лучше откалиброванные средние значения, чем запрос одного числа. Также позволяет маршрутизацию по уверенности в Фазе 2 (широкий диапазон = направить на ревью).

**Калибровочные якоря — начальные значения, не истина.**

Числовые якоря в системном промпте (50-100 строк/час, 0.5ч за 50 переименований и т.д.) — экспертные эвристики, согласованные с индустриальной практикой, но не взятые из опубликованных исследований. Их НЕОБХОДИМО рассматривать как начальные параметры, подлежащие пересмотру после первых 10-20 GT-валидаций. Реализация должна читать их из `calibration_config` (env или файл), а не хардкодить в шаблон промпта.

## 5. LLM-as-Judge: проверка правдоподобности (Фаза 1.5)

Второй вызов для проверки правдоподобности. Дешёвый (класс Haiku), ловит очевидные выбросы:

```
COMMIT SUMMARY: {message}, {fc} files, +{la}/-{ld}
ESTIMATE: {low}-{mid}-{high}h, confidence={confidence}

PLAUSIBILITY RULES:
- Single-module feature: rarely >40h
- Config/tooling migration: rarely >16h
- Bulk rename/move: rarely >4h
- Full-system new feature with tests: can reach 60-80h

Verdict: PLAUSIBLE, OVER, or UNDER?
Return: {"verdict": "...", "suggested_mid": N, "reason": "..."}
```

При вердикте OVER или UNDER: принять `suggested_mid` судьи напрямую (или смешать с весом 0.7 в пользу судьи). Судья явно применяет правила правдоподобности — если он говорит "tooling-коммит, не должен превышать 16ч", эта ограничительная коррекция надёжнее наивного усреднения двух оценок.

## 6. План реализации

### Фаза 1: MVP холистического оценщика (этот спринт)

**Цель**: Заменить v2 Branch B + combine одним холистическим вызовом. Целевой MAPE <50%.

**Новый код:**
1. `compute_entropy(file_info)` — энтропия Шеннона для распределения изменений
2. `compute_metadata_block(file_info, message, language, ...)` — собирает все P0/P1 фичи в структурированный текст
3. `estimate_holistic_v3(metadata_block, call_ollama_fn)` — один LLM-вызов с богатыми метаданными

**Обновлённый код:**
4. `_run_fd_v2()` — заменить поток Branch B/A/holistic/combine на `estimate_holistic_v3()`
5. Убрать добавление `heuristic_total` — LLM видит его как метаданные, сама определяет вес

**Сохраняется (используется как источники метаданных):**
- `adaptive_filter()` — классифицирует файлы в SKIP/HEURISTIC/SUBSTANTIVE
- `build_clusters()` — группирует файлы для структурного контекста в промпте
- `classify_file_tier()` — поимённая классификация
- Scaffold-детектор, проверки дешёвых сигналов — ранние выходы перед v3 gate

**Удаляется:**
- `estimate_branch_b()` — покластерная LLM-оценка
- `estimate_branch_a()` — оценка полным диффом в одном вызове
- `combine_estimates()` — комбинирование двух сигналов
- `call_openrouter_large()` — отдельный API-вызов мощной модели

**Валидация:**
- Запуск на всех 10 GT-коммитах из `docs/ground-truth-request.md`
- **Только zero-shot** — НЕ использовать GT-коммиты как few-shot примеры (data leakage)
- Вычислить baseline: MAPE оценщика на основе только `heuristic_total` для сравнения
- Критерий выпуска: MAPE v3 < 50% И MAPE v3 < MAPE baseline (должен побить наивную формулу)

### Фаза 2: Калибровка + судья (следующий спринт, требует 20+ GT-коммитов)

1. LLM-as-Judge проверка правдоподобности (Haiku, $0.001)
2. Предварительный вызов для классификации intent (Haiku, $0.001)
3. Мультипликативная коррекция per-commit-type из GT-данных
4. Ансамбль с вариацией температуры (3 вызова при T=0.0/0.3/0.7, берём медиану) для низкой уверенности
5. Пересмотр калибровочных якорей в системном промпте по результатам анализа ошибок Фазы 1

### Фаза 3: Адаптация (будущее, требует 30+ GT-коммитов)

1. Хранилище feature-векторов из накопленных GT-данных
2. Байесовский prior (эвристическая оценка) + LLM posterior с обучаемыми весами

## 7. Результаты эксперимента (Фаза 1)

Эксперимент: 2026-03-27. Скрипт: `packages/server/scripts/pipeline/experiment_v3.py`. 5 моделей x 10 GT-коммитов, zero-shot, промпт только из метаданных (~1500 input-токенов на вызов).

### 7.1 Сравнение моделей

| Модель | MAPE | MdAPE | В диапазоне | В пределах 2x | Bias (сред.) | Over/Under | $/коммит |
|--------|------|-------|-------------|---------------|-------------|------------|----------|
| **Opus 4.6** | **26.0%** | **20.0%** | **6/10** | **10/10** | +7.0ч | 7O/2U | $0.036 |
| **Qwen3 Coder+** | **32.1%** | **25.2%** | **6/10** | **10/10** | -1.9ч | 4O/5U | $0.001 |
| Sonnet 4.6 | 80.2% | 46.2% | 4/10 | 8/10 | +30.1ч | 8O/1U | $0.007 |
| Haiku 4.5 | 131.7% | 103.9% | 2/10 | 7/10 | +40.4ч | 10O/0U | $0.002 |
| GPT-5.3 Codex | 170.0% | 170.0% | 0/5* | 2/5* | +57.8ч | 5O/0U | $0.009 |

*GPT-5.3 Codex: 5/10 вызовов провалились (ошибки парсинга JSON / NoneType). Метрики только по успешным.

### 7.2 Ключевые выводы

1. **Цель Фазы 1 превышена**: Opus MAPE 26% и Qwen MAPE 32% — оба значительно ниже порога <50%. Baseline v2 был 80.4%.
2. **Метаданные работают**: Без единой строки кода в промпте топовые модели достигают точности, сопоставимой с экспертной оценкой (30-50% MAPE по литературе).
3. **Стоимость пренебрежимо мала**: Промпт v3 ~1500 токенов input. При $0.001/коммит (Qwen) даже 100 больших коммитов = $0.12 на заказ.
4. **Qwen3 Coder+ — чемпион по value**: В 30x дешевле Opus, всего на 6пп хуже по MAPE, лучший калиброванный bias (-1.9ч, почти ноль).
5. **Систематическая переоценка у слабых моделей**: Haiku (10/10 over), Sonnet (8/1 over), GPT (5/0 over). Только Qwen имеет сбалансированный bias.
6. **GPT-5.3 ненадёжен для structured output**: 50% failure rate на JSON-ответах.

### 7.3 Решение: модель для продакшена

- **Пилотная фаза**: Qwen3 Coder+ (`qwen/qwen3-coder-plus`) — лучшее соотношение цена/качество, сбалансированный bias, 10/10 в пределах 2x.
- **Enterprise-тир** (будущее): Opus 4.6 — лучшая точность, но в 30x дороже. Зарезервирован для премиум-планов.

| Задержка | 80-110с | **~4с** | <15с | <20с |

## 8. Анализ рисков

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| LLM недооценивает сложные фичи, которых не видит в диффе | Средняя | Высокое | Judge-вызов ловит выбросы; флаг низкой уверенности для ревью |
| Переобучение на GT-наборе (всего 10 коммитов) | Высокая | Среднее | Leave-one-out кросс-валидация; сбор дополнительных GT-данных |
| LLM игнорирует метаданные heuristic_total, недооценивает тривиальную работу | Средняя | Среднее | Валидировать на config-heavy коммитах; добавить floor при необходимости |
| Изменения API модели влияют на калибровку | Низкая | Среднее | Зафиксировать версию модели; перекалибровать на GT после смены модели |
| Фичи энтропии/распределения не улучшают MAPE | Средняя | Низкое | Система работает без них; откатить если улучшение не измерено |
| LLM возвращает невалидный JSON | Средняя | Низкое | Принудительная схема через response_format; regex fallback |

## 9. Открытые вопросы и решения по ревью

### Решены после экспертного ревью:

1. **Floor для heuristic_total** — **Не нужен в Фазе 1.** Показать LLM числа по формуле, дать модели решить. Вводить floor только если GT-валидация выявит систематическую недооценку для config-heavy коммитов.

3. **Нормализация энтропии** — **Сырое значение + максимум для контекста.** `Entropy: 3.42 (max=7.09; умеренно распределено)`. Нормализация к [0,1] теряет интерпретируемость для LLM.

4. **Few-shot в Фазе 1** — **Нет. Только zero-shot.** Использование 10 GT-коммитов одновременно как few-shot примеры и как evaluation set = data leakage. Few-shot вводится в Фазе 2 при наличии 20+ GT-коммитов и возможности разделить train/test.

5. **Сосуществование v2/v3** — **Параллельный запуск через `FD_V3_ENABLED=true`.** Единственный способ получить честное сравнение на одних и тех же коммитах.

### Решены после эксперимента Фазы 1 (2026-03-27):

2. **Выбор модели** — **Qwen3 Coder+ для пилота, Opus 4.6 для enterprise.** Эксперимент показал: Qwen достигает MAPE 32% при $0.001/коммит (в 30x дешевле Opus с 26% MAPE). Sonnet/Haiku/GPT систематически переоценивают. Qwen имеет лучший калиброванный bias (-1.9ч). Модель для Judge (Фаза 2): Haiku за $0.002/вызов — его переоценка приемлема для проверки правдоподобности (ловля выбросов, не точность).

6. **Определение effective churn** — **Только содержательные (LLM-тир) файлы.** Промпт эксперимента использует `effective_la`/`effective_ld` из LLM-файлов `adaptive_filter()`. Часы heuristic-тира передаются отдельно как метаданные `heur_total`. Это исключает двойной счёт и даёт LLM чёткий сигнал: что требует человеческой оценки vs что считается по формуле.
