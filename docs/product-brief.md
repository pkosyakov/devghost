# DevGhost — Product Brief

> Документ для планирования продвижения продукта.
> Дата: 2026-02-24 | Стадия: MVP / бета

---

## 1. Суть продукта

**DevGhost** — платформа аналитики эффективности разработки. Анализирует Git-репозитории с помощью ИИ, оценивает реальные трудозатраты по каждому коммиту и рассчитывает справедливую стоимость работы разработчиков.

Одной фразой: **"ИИ-аудитор, который читает код и считает, сколько это стоило на самом деле."**

---

## 2. Проблема

### Для компаний-заказчиков
- Невозможно оценить, насколько справедлива стоимость разработки. Подрядчик выставил 500 часов — но сколько из них реальная работа?
- Строки кода и количество коммитов ничего не говорят о сложности. Переименование файла и архитектурный рефакторинг — это один коммит, но разница в трудозатратах в десятки раз.
- Нет объективного инструмента для проверки. Приходится доверять или нанимать технического аудитора.

### Для тимлидов и CTO
- Нет ясной картины загрузки команды. Кто перегружен, кто недозагружен — видно только по ощущениям.
- Сложно обосновать решения по ставкам и бонусам. Нужны данные, а не субъективные оценки.
- Планирование вслепую: непонятно, сколько реально стоит фича или спринт.

### Для фрилансеров и подрядчиков
- Сложно доказать клиенту, что работа стоит запрошенных денег. "Я переписал 50 строк, но это заняло 3 дня" — звучит неубедительно без контекста.
- Нет инструмента, который покажет объективную картину трудозатрат и подтвердит адекватность ставки.

### Общая боль
Существующие метрики разработки (строки кода, количество PR, время цикла) измеряют **активность**, а не **усилия**. Они не отвечают на вопрос: "Сколько когнитивных усилий стоит эта работа?"

---

## 3. Решение

DevGhost подключается к Git-репозиториям, извлекает коммиты и пропускает каждый дифф через языковую модель (LLM), которая оценивает трудозатраты в часах. Затем система агрегирует эти оценки по разработчикам и периодам, рассчитывает метрики продуктивности и формирует итоговую стоимость работы.

### Как это работает (без технических деталей)

1. **Подключаешь репозитории** — свои через GitHub OAuth или любые публичные по ссылке
2. **Система извлекает разработчиков** — находит всех авторов коммитов, автоматически объединяет дубликаты (один человек может коммитить с разных email)
3. **ИИ анализирует каждый коммит** — читает изменения кода и оценивает, сколько часов реальной работы за ними стоит
4. **Получаешь результат** — дашборд с метриками по каждому разработчику: трудозатраты, продуктивность, стоимость

### Ключевое отличие

DevGhost не считает строки и не измеряет скорость закрытия тикетов. Он **читает код** и оценивает когнитивную сложность работы — так же, как это сделал бы опытный разработчик при code review, только автоматически и по всей истории репозитория.

## 4. Целевая аудитория

### Сегмент 1: Компании-заказчики разработки

**Кто:** Бизнес, который заказывает разработку у подрядчиков или аутсорсинговых компаний.

**Боль:** Получают счёт за 800 часов работы и не могут проверить, насколько это адекватно.

**Ценность DevGhost:** Независимый ИИ-аудит. Загружают репозиторий проекта, получают объективную оценку трудозатрат каждого разработчика. Могут сравнить с выставленным счётом.

**Сценарий:** "Подрядчик выставил $40,000 за 3 месяца работы. Мы прогнали репозиторий через DevGhost и увидели, что реальные трудозатраты команды из 4 человек — 320 часов, а не 500. Ghost % двух разработчиков — ниже 70%. Обсудили это с подрядчиком и пересмотрели условия."

### Сегмент 2: Тимлиды и CTO

**Кто:** Технические руководители команд разработки.

**Боль:** Нужно понимать реальную загрузку каждого разработчика, обосновывать кадровые решения данными, планировать ресурсы.

**Ценность DevGhost:** Объективная картина продуктивности команды. Ghost % каждого разработчика, распределение усилий по дням, выявление overhead (встречи, код-ревью, дебаг — время, которое не отражается в коммитах).

**Сценарий:** "При ретроспективе спринта видно, что Ghost % одного разработчика упал с 95% до 60%. Оказалось, он 40% времени тратил на менторство джуниоров, что не отражалось в коммитах. Перераспределили нагрузку."

### Сегмент 3: Фрилансеры и подрядчики

**Кто:** Разработчики, которые работают по контрактам и хотят обосновать свою стоимость.

**Боль:** Клиенты не понимают, почему "50 строк кода" стоят $2000.

**Ценность DevGhost:** Прозрачный отчёт о трудозатратах, который можно показать клиенту. ИИ-оценка каждого коммита подтверждает, что маленький дифф может означать дни исследования и сложной работы.

**Сценарий:** "После каждого спринта отправляю клиенту ссылку на отчёт DevGhost. Он видит breakdown по дням и коммитам, понимает, за что платит. С тех пор ни одного спора по часам."

## 5. Ключевые возможности

### 5.1. Подключение репозиториев

Три способа добавить репозиторий для анализа:

- **Мои репозитории** — подключаешь GitHub аккаунт через OAuth, видишь все свои репозитории (включая приватные), выбираешь нужные
- **Публичный репозиторий** — вводишь ссылку на любой публичный репозиторий (owner/repo), система подтягивает его
- **Explore (поиск)** — продвинутый поиск по GitHub: фильтры по языку, количеству контрибьюторов, активности (коммиты/неделя), звёздам. Помогает найти похожие проекты для бенчмарка

### 5.2. Умная дедупликация разработчиков

Один разработчик может коммитить с разных email (рабочий, личный, старый). Система:

- Автоматически находит и объединяет профили с высокой уверенностью
- Предлагает ручное объединение для неоднозначных случаев
- Позволяет исключить ботов и сервисные аккаунты из анализа

### 5.3. ИИ-оценка трудозатрат

Ядро продукта. Для каждого коммита:

- LLM читает дифф (изменения кода)
- Оценивает трудозатраты в часах с учётом сложности, контекста и типа работы
- Различает механическую работу (rename, auto-format) и интеллектуальную (архитектура, отладка, алгоритмы)

Поддерживается два провайдера LLM:
- **Ollama** — локальная модель, бесплатно, данные не покидают машину
- **OpenRouter** — облачные модели (GPT, Claude и др.), платно, выше качество оценки

### 5.4. Настройка периода анализа

Гибкий выбор того, что анализировать:

- **Всё время** — полная история репозитория
- **Диапазон дат** — конкретный период с пресетами (квартал, полгода, год)
- **Последние N коммитов** — 50, 100, 200 или 500 последних коммитов

Перед запуском анализа показывается preview: сколько коммитов попадает в выбранный период и сколько разработчиков в них участвовало.

### 5.5. Дашборд результатов

После завершения анализа:

**KPI-карточки:**
- Средний Ghost % по команде
- Количество разработчиков
- Общее число коммитов
- Суммарные рабочие дни

**Визуализации Ghost %:**
- Bubble chart — размер пузыря = трудозатраты, оси: коммиты vs Ghost %
- Strip chart — горизонтальное распределение Ghost %
- Heatmap — тепловая карта по разработчикам

**Таблица разработчиков:**
- По каждому: коммиты, рабочие дни, трудозатраты (часы), среднее в день, overhead, доля (%), Ghost %
- Раскрытие строки показывает **дневной таймлайн** — цветовая шкала нагрузки по дням с детализацией до коммита
- Настраиваемая доля участия (%) — автоматический расчёт или ручная корректировка

**Фильтрация по периодам:**
- Переключение между All Time, по годам, по месяцам
- Все метрики пересчитываются динамически

### 5.6. Бенчмарк моделей

Возможность прогнать один и тот же анализ на разных LLM-моделях и сравнить результаты. Позволяет выбрать оптимальное соотношение цены и качества оценки.

### 5.7. Прозрачность стоимости анализа

Перед запуском показывается:
- Какая LLM-модель будет использована
- Стоимость за коммит (если облачная модель)
- Общая расчётная стоимость анализа

После завершения:
- Реальная стоимость (токены, вызовы LLM, итог в $)

### 5.8. Детекция overhead

Система выявляет overhead — время, которое тратится на работу, не отражённую в коммитах (встречи, планирование, код-ревью, дебаг без результата). Отображается на дневном таймлайне красными маркерами.

## 6. Путь пользователя

### Быстрый старт (5 минут)

```
Регистрация → Демо-режим → Просмотр готового отчёта
```

Пользователь может сразу увидеть продукт в действии на демо-данных: 2 репозитория, 5 разработчиков, 10 коммитов с рассчитанными метриками.

### Полный цикл (15-30 минут для первого анализа)

```
Регистрация
    ↓
Подключение GitHub (опционально, даёт доступ к приватным репо)
    ↓
Создание анализа → Выбор репозиториев → Настройка периода
    ↓
Извлечение разработчиков → Проверка дедупликации → Исключение ботов
    ↓
Просмотр оценки стоимости анализа → Запуск
    ↓
Ожидание (реалтайм-прогресс: X/Y коммитов, логи пайплайна)
    ↓
Дашборд с результатами → Исследование метрик → Экспорт/обсуждение
```

### Повторное использование

- Повторный анализ того же набора репозиториев — в один клик
- Изменение периода — редактирование scope без пересоздания заказа
- Новый анализ — создание нового заказа с другими репозиториями

## 7. Ключевая метрика: Ghost %

### Что это

Ghost % (процент "призрака") — отношение оценённой продуктивности разработчика к ожидаемой загрузке. Название отражает идею: сколько "невидимой" работы стоит за видимым результатом.

### Как считается

1. ИИ оценивает трудозатраты по каждому коммиту (в часах)
2. Суммируются по разработчику за период
3. Сравниваются с ожидаемой загрузкой (рабочие дни x доля участия в проекте)
4. Результат в процентах

### Шкала

| Ghost % | Значение | Интерпретация |
|---------|----------|---------------|
| **> 100%** | Перевыполнение | Разработчик выдаёт больше, чем ожидается при данной загрузке. Может означать высокую продуктивность, но также — риск выгорания |
| **80–100%** | Норма | Здоровая продуктивность, результат соответствует ожиданиям |
| **60–80%** | Ниже ожиданий | Возможные причины: overhead (встречи, ревью), онбординг, сложные задачи с длинным циклом |
| **< 60%** | Значительно ниже | Требует внимания: может быть блокер, неэффективные процессы или несоответствие роли |

### Цветовая индикация

- Зелёный — нормальный или высокий Ghost %
- Жёлтый — требует внимания
- Красный — значительно ниже ожиданий
- Серый — недостаточно данных для оценки

### Важные нюансы

- Ghost % не показатель "хорошести" разработчика. Это инструмент диагностики.
- Низкий Ghost % не всегда плохо: человек может тратить время на менторство, архитектурные решения, код-ревью — что не отражается в коммитах.
- Метрику нужно интерпретировать в контексте: роль, тип задач, процессы в команде.

## 8. Конкурентные преимущества

### Vs. существующие инструменты аналитики (LinearB, Jellyfish, Waydev, Pluralsight Flow)

| Аспект | Конкуренты | DevGhost |
|--------|-----------|----------|
| **Что измеряют** | Proxy-метрики: cycle time, PR throughput, строки кода, частота деплоев | Когнитивные трудозатраты через ИИ-анализ кода |
| **Как считают** | Статистический анализ Git/Jira метаданных | LLM читает дифф каждого коммита и оценивает сложность |
| **Точность оценки усилий** | Строки кода ≠ усилия. 1 строка фикса может быть сложнее 500 строк boilerplate | Различает механическую и интеллектуальную работу |
| **Стоимость** | $15–50/dev/месяц, минимум 10 разработчиков | LLM-модель по выбору (от бесплатной локальной до облачной) |
| **Приватность данных** | Данные на серверах вендора | Можно запустить полностью локально (Ollama) |
| **Что нужно для старта** | Интеграция с Jira, GitHub, CI/CD | Только Git-репозиторий |

### Уникальные преимущества DevGhost

1. **ИИ-первый подход** — не прокси-метрики, а прямая оценка кода. Модель понимает, что `rename` и `рефакторинг архитектуры` — это разные трудозатраты, даже если дифф одного размера.

2. **Независимость от процессов** — не нужен Jira, не нужны тикеты, не нужен CI/CD. Достаточно Git-репозитория. Работает с любым процессом разработки.

3. **Приватность** — можно запустить полностью локально. Код не покидает машину. Критично для компаний с NDA или чувствительным кодом.

4. **Гибкость LLM** — выбор между бесплатной локальной моделью и платными облачными. Бенчмарк позволяет сравнить качество оценок.

5. **Финансовый фокус** — не просто метрики "для понимания", а конкретные цифры: часы, ставки, стоимость. Инструмент для финансовых решений.

6. **Низкий порог входа** — демо-режим показывает продукт без настройки. Публичные репозитории можно анализировать без GitHub OAuth.

## 9. Текущее состояние продукта

### Стадия: MVP / бета

### Что готово и работает

- Полный цикл анализа: от подключения репозитория до дашборда с результатами
- GitHub OAuth интеграция (приватные репозитории)
- Анализ публичных репозиториев без авторизации
- ИИ-оценка трудозатрат (Ollama и OpenRouter)
- Дедупликация разработчиков (автоматическая + ручная)
- Дашборд с визуализациями (3 типа графиков, таблица, дневной таймлайн)
- Бенчмарк LLM-моделей
- Настройка периода анализа (всё время, диапазон дат, последние N коммитов)
- Демо-режим
- Авторизация (email + GitHub OAuth)
- Кеширование (клонированные репо, диффы, ответы LLM)

### В активной разработке

- **Explore Search v2** — продвинутый поиск репозиториев с фильтрами по активности, языку, размеру команды
- Улучшение точности ИИ-оценки

### Не реализовано (потенциальные направления)

- Экспорт отчётов (PDF, CSV)
- Интеграция с GitLab, Bitbucket
- Командные дашборды (несколько заказов в одном view)
- API для внешних интеграций
- Мобильное приложение (заложена структура, но не реализовано)
- Уведомления и автоматические отчёты
- Сравнение проектов между собой

## 10. Бизнес-модель

Модель монетизации на стадии определения. Возможные направления:

### Вариант A: SaaS подписка
- Бесплатный tier: публичные репозитории, локальная LLM
- Платный tier: приватные репозитории, облачные LLM, расширенная аналитика
- Enterprise: self-hosted, SSO, аудит

### Вариант B: Usage-based
- Оплата за количество проанализированных коммитов
- Прозрачная модель: платишь только за то, что используешь

### Вариант C: Отчёт-как-услуга
- Разовый аудит: загрузил репозиторий — получил отчёт — заплатил
- Подходит для компаний-заказчиков, которым нужна одноразовая проверка

### Текущее состояние
Продукт self-hosted, пользователь сам оплачивает LLM-вызовы (если использует облачную модель). Базовый функционал с локальной LLM полностью бесплатен.

## 12. Терминология продукта

| Термин | Значение |
|--------|----------|
| **Ghost %** | Основная метрика — отношение оценённой продуктивности к ожидаемой загрузке |
| **Заказ (Order)** | Единица анализа: набор репозиториев + период + разработчики |
| **Трудозатраты (Effort)** | Оценённое ИИ количество часов работы за коммитом |
| **Overhead** | Время работы, не отражённое в коммитах (встречи, ревью, дебаг) |
| **Доля (Share %)** | Процент занятости разработчика на проекте |
| **Эффективная ставка** | Расчётная стоимость часа работы с учётом Ghost % |
| **Дедупликация** | Объединение профилей одного разработчика с разных email |
| **Explore** | Поиск репозиториев на GitHub по характеристикам для бенчмарка |
